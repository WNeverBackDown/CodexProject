# 智慧互联测试平台策略

## 第一阶段：离线可用的小闭环

- 用 ADB 做设备发现、属性采集、蓝牙状态采集和 logcat 采集。
- 用 pytest 编排 smoke test 和无设备可跑的本地验证。
- 用本地规则诊断器分析失败日志，输出风险分类、证据行、建议和置信度。
- 用 Markdown 报告把测试证据和诊断结果汇总起来，便于贴到缺陷单。
- 默认开启敏感信息脱敏，避免报告中泄露 MAC、VIN、手机号、账号、设备序列号等数据。

## 第二阶段：多设备和车机场景自动化

- 用 Appium 自动化系统设置、车载互联 App、蓝牙配对弹窗和授权弹窗。
- 用 Mobly 编排多设备场景，例如手机 A 与手机 B、手机与车机、手机与 BLE 外设。
- 引入 HCI snoop、车机日志、截图、录屏和 Wireshark 摘要，做跨证据关联。
- 将常见车机互联 SOP 固化为可复测场景：首次连接、断连重连、手机重启恢复、来电/通知打断、弱网/弱信号。

## 第三阶段：可插拔 AI 引擎

- 保留 `local_rules` 作为默认引擎，确保公司内网或无网环境可用。
- 增加 `local_http`、`company_gateway`、`openai` 等引擎适配器时，只替换诊断层，不改采集和报告。
- 用历史缺陷库生成高风险测试矩阵，并根据失败模式推荐最小复测集。
- 将报告同步到 Jira、禅道、TestRail 或公司内部质量平台。

## 工具价值方向

- 日志归因提效：把多源日志中的关键失败信号先筛出来，减少人工翻日志时间。
- 用例设计提效：根据功能域和风险点快速生成覆盖矩阵。
- 缺陷提交提效：自动整理复现步骤、证据、疑似根因和复测建议。
- 回归选择提效：根据失败分类推荐必须回归的最小场景集合。
- 新人上手提效：把隐性的测试经验沉淀为规则、场景和报告模板。
